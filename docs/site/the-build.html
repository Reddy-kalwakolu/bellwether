Day 00 · The idea

A job posting, and a problem worth solving

It starts with a Netflix job posting — requisition AJRT30201, Staff AI Engineer on the Ads Platform team. Read past the boilerplate and the posting is describing a specific, current problem: engineering teams can now ship code far faster than they can verify it.

AI writes the code quickly. It also writes plausible-looking code that is subtly wrong, and nobody notices until production does. The posting asks for someone who can make a team fast without accumulating that debt.

Most portfolio projects respond to that by demoing an impressive-looking agent. An agent demo proves almost nothing — anyone can find a prompt that works once on camera. So this project takes the opposite bet.

The bet

Every capability ships with a number

Not "here is an agent that reviews pull requests" but "here is an agent that caught 7 of 10 deliberately planted bugs, here is the eval that measured it, and here is the run where it scored 4."

That's harder to build and much harder to fake. It is also the only version of this project that a Staff engineer would find interesting rather than familiar.

The thing nobody builds first

There's a problem with AI-operations demos. To show an agent diagnosing an outage, you need an outage. Most demos fake one — a JSON file of pretend logs, a chart drawn from invented numbers.

An agent tested against a fake failure can only ever find the fake. The evaluation measures the fixture, not the agent.

So BELLWETHER is deliberately two systems, and the unglamorous one gets built first.

The substrate is a miniature version of the kind of system Netflix's ads team actually runs. It exists so the AI layer has something genuine to operate on. When an agent later diagnoses a broken deployment, the deployment will really be broken.

Days 1 to 5 build the substrate. Everything after that is the AI layer.

Day 01 · Foundation

The boring day that decides the next twenty-nine

Day 1 produces nothing a user could touch. It sets up the workshop: how code gets checked, how the database starts, how anyone knows whether the thing is working.

Skipping this is the single most common way a 30-day build collapses around day nine.

The quality gates, installed before the code

Four tools run on every single change, and all four must pass:

ToolWhat it doesWhy it's non-negotiable
ruffCatches unused imports, bad patterns, style driftStyle arguments are a waste of a day; a tool ends them
ruff formatReformats every file identicallyDiffs show meaning, not whitespace
mypy --strictVerifies every value is the type claimedCatches a whole class of bug before the code runs
pytestRuns the test suiteThe only evidence that anything works

--strict matters more than it sounds. Normal type checking is advisory; strict mode refuses to let anything be untyped. It's uncomfortable for about two days and then it silently prevents dozens of bugs. Later, when AI agents start generating code for this repo, these four tools become the fence — generated code that fails them never lands.

The first real decision

The system needs a database, a cache, a metrics collector and a dashboard, all started reliably. The industry-default answer in 2026 is Kubernetes.

ADR-0001

Docker Compose, not Kubernetes

Kubernetes is built for hundreds of services across many machines, surviving hardware failure and traffic spikes. This is four services on one laptop.

Kubernetes would add days of configuration and prove nothing about AI engineering — the actual subject. Compose starts everything with one command and stays legible.

Choosing infrastructure proportionate to the problem is the senior signal. Reaching for the biggest tool available is the opposite.

Trigger to revisit: if this ever needs to run across more than one machine.

Every decision on this project is written down like that — as an Architecture Decision Record. Each one names the alternatives that were rejected and, crucially, the trigger that would change the answer. A decision without a trigger is just an opinion that hardens into folklore.

Small detail, saved hours

Postgres was published on port 5433 instead of the standard 5432, and Redis on 6380 instead of 6379. Any other database already running on the machine would otherwise silently collide, producing failures that look like code bugs and are not.

Day 1 also shipped the running doc — a live HTML page with the architecture, a 30-day tracker, and an eval scoreboard whose numbers were all empty. Publishing empty cells on day one is a commitment: those cells are the whole point of the series.

Day 02 · campaign-service

Where the advertisers live

First real service. It stores campaigns — an advertiser's instructions about when and to whom their advert may be shown.

The domain vocabulary here is deliberate. These are the exact terms the Netflix posting uses:

TermIn plain English
targetingWho may see this — countries, device types, content ratings
frequency capThe most times one viewer may see it in a day
brand safetyContent this advertiser refuses to appear beside
pacingSpreading a daily budget across the day instead of by 9am
creativeThe actual video file, with its duration
flight windowThe dates the campaign runs between

    C["Campaign<br/>budget · daily budget<br/>frequency cap<br/>flight window"]
    T["Targeting<br/>countries<br/>device types<br/>content ratings"]
    B["Brand safety<br/>excluded categories"]
    CR["Creative<br/>asset URL<br/>duration"]
    C --> T
    C --> B
    C -->|"one to many"| CR
    C --> PG[("Postgres")]

Money is never a decimal

Budgets are stored as whole numbers of micros — millionths of a currency unit. Ten dollars is 10000000.

This looks awkward and prevents a real disaster. Computers cannot represent 0.1 exactly in the decimal format most languages default to. Add it ten times and you get 0.9999999999999999. Do that across millions of ad impressions and the books stop balancing. Whole numbers never drift.

ADR-0002

Build the tables directly, no migration tooling — for now

Normally, changing a database's shape needs a migration: a versioned script so an existing database can be upgraded without losing data. The standard tool is Alembic.

At this point there is one service writing, no data anyone depends on, and a schema that will change every single day through Day 5. Migrations written against a shape that churns daily are noise pretending to be safety.

Trigger to revisit: the first time a second service reads these tables.

Failures have a shape

Every error — a bad request, a missing campaign, anything — returns the same envelope:

flowchart LR
    OK["Success<br/>the object you asked for"]
    ERR["Failure<br/>{ error: { code, message } }"]
    R["Any request"] --> OK
    R --> ERR
    ERR -.->|"never a bare 500"| N["a caller can always<br/>parse what went wrong"]

A generic "500 Internal Server Error" tells a caller nothing. Every failure here is typed and explained. That standard gets violated on Day 5, gets caught, and gets fixed — which is exactly what writing a standard down is for.

Every request also emits one line of structured JSON to the logs, carrying the service name, the endpoint, and how many milliseconds it took. Not prose — data. On Day 16, ops agents will read these without a single regular expression.

Day 2 ended at 23 tests passing.

Day 03 · ad-decision-service

The twenty milliseconds that matter

This is the heart of any ads platform. A viewer reaches an ad break. Something must decide, in milliseconds, which advert to play — or that none is suitable.

The request carries who is watching and what slot is available. The answer is one advert, or a no-fill with a reason.

Six filters, in order

Every campaign in the system is a candidate. Each filter eliminates some, and the order is chosen so the cheapest checks run first:

flowchart TB
    REQ["Ad request<br/>member + slot"] --> F1
    F1["1 · Eligibility<br/>is it active and in flight?"] --> F2
    F2["2 · Targeting<br/>right country, device, rating?"] --> F3
    F3["3 · Brand safety<br/>does this content offend the advertiser?"] --> F4
    F4["4 · Frequency cap<br/>has this viewer seen it enough?"] --> F5
    F5["5 · Budget pacing<br/>has it spent its allowance for now?"] --> F6
    F6["6 · Creative fit<br/>is there a video of the right length?"] --> WIN
    WIN["Winner: most daily budget remaining"] --> OUT["Filled ✓"]
    F1 -.->|"expired"| NF["No-fill,<br/>with a named reason"]
    F2 -.->|"targeting_mismatch"| NF
    F3 -.->|"brand_safety_excluded"| NF
    F4 -.->|"frequency_capped"| NF
    F5 -.->|"pacing_throttled"| NF
    F6 -.->|"no_creative_fit"| NF
Design rule

Named reasons, never booleans

A filter that returns true or false tells you nothing at 2am. Every rejection here carries a name: pacing_throttled, targeting_mismatch, frequency_capped.

Those names become a metric label, a log field, and a line in the API response. On Day 4 they become a Grafana chart where a fill-rate collapse is diagnosed by reading which band grew. On Day 18 they are the evidence an RCA agent cites.

Every response also carries a full trace: which campaigns lost and to which rule. Not for debugging — for the ops agents that arrive in Level 3.

Two kinds of memory

Frequency capping needs to know how many times a viewer has already seen an advert today. That count is written on every single impression, is per-viewer, and is worthless after midnight.

ADR-0003

Redis holds the decision state, Postgres holds the truth

A count that expires at midnight is a counter with a timer, not a permanent record. Redis holds those under keys that delete themselves after 48 hours. Postgres remains the system of record.

Trigger to revisit: Day 4's event-service, when impressions become permanent and spend becomes billable.

Note the discipline here: the decision service reads campaigns through campaign-service's public API, never its database tables. That is precisely what keeps ADR-0002's Alembic trigger — "the first time a second service reads these tables" — deliberately unfired. Two ADRs, written a day apart, holding each other in place.

Pure rules, injected state

All six filters live in one file with no input or output of its own. It receives the data it needs and a clock. That's why the entire test suite runs with Docker switched off, in about five seconds.

What reality found

Two services, one global counter, instant collision

Prometheus keeps one registry per running process. Both services defined a metric called http_requests_total, which was fine until the test suite imported both at once — then it crashed with a duplicate-metric error.

Fixed by defining the shared metrics once in substrate/shared/observability.py, with a service label separating them. The plan hadn't foreseen it; running the tests did.

Day 3 ended at 51 tests passing.

Day 04 · event-service + dashboards

Counting things exactly once

An advert played. That's an impression, and it costs the advertiser money. Someone clicked it. That's a click. Both must be recorded.

The hard part is not storing them. It's storing them exactly once.

Why counting is hard

Networks are unreliable. A player reports an impression, the response gets lost, the player retries. Now the same impression arrives twice. Count both and the advertiser is billed twice.

The obvious fix is to check first: have I seen this before? No? Then save it. That fix is broken, and the reason is worth understanding:

sequenceDiagram
    participant A as Report A
    participant B as Report B, a retry
    participant DB as Database
    Note over A,B: same impression, same moment
    A->>DB: seen event-123?
    DB-->>A: no
    B->>DB: seen event-123?
    DB-->>B: no
    A->>DB: save it
    B->>DB: save it
    Note over DB: counted twice
ADR-0004

Let the database refuse it

The caller generates the event's ID, and that ID is the table's primary key. A primary key is unique by definition, enforced by the database itself.

So both reports just try to save. One wins. The other is rejected by the database, and the service answers 200 duplicate. There is no gap between checking and writing, because there is no check.

A duplicate returns success, not an error — from the caller's point of view the event is recorded. An error would train clients to retry harder at exactly the wrong moment.

Server-generated IDs were never an option: every retry would get a fresh ID and therefore become a new impression. The caller has to own the ID for any of this to work.

Totals are computed by scanning the events table on demand, not maintained in a separate running-total table. A running total is a second copy of the truth, and two copies drift the first time anyone corrects historical data. The trigger is written down: when that scan exceeds ~200ms under load.

The trigger from yesterday fired

ADR-0003 said Redis would hold spend until impressions became durable. Today they did. The decision narrowed rather than reversed:

flowchart LR
    R[("Redis<br/>what the pacer believed<br/>at decision time")]
    P[("Postgres<br/>what was actually<br/>delivered")]
    R -.->|"the gap between them"| G["reporting loss<br/>— a dashboard panel,<br/>not a bug"]
    P -.-> G

They now disagree by design, and the disagreement is the interesting number.

Dashboards that live in the repository

Two Grafana dashboards, defined as files in version control rather than clicked together in a browser:

  • substrate-health — are services up, request rates, error ratios, p95 latency
  • ads-delivery — fill rate, why candidates lost, events, click-through rate, spend

The "why candidates lost" panel is the payoff for Day 3's named reasons. It's a stacked chart, one band per rejection reason. When fill rate collapses, you read the answer off whichever band grew.

A test guards them — every panel must have a query, point at the right data source, and graph a metric a service actually emits. It runs with Docker off.

What reality found

An unnamed data source, and every panel loads empty

Grafana generates a random internal ID for its data connection at first startup. The dashboards referenced that ID. Restart, new ID, and every panel silently renders blank against a connection that no longer exists.

Fixed by pinning the ID explicitly — plus a test asserting the dashboards and the data source still agree. The plan missed it; the live stack surfaced it in thirty seconds.

Day 4 ended at 87 tests passing, four services scraped by Prometheus.

Day 05 · traffic-simulator + the gate

Breaking it on purpose, for real

Four services exist but nothing uses them. Day 5 builds a simulator that generates realistic viewers, requests adverts, and reports impressions and clicks back — continuously.

Then it breaks things. Five failure modes, switchable live over an API:

ScenarioWhat it doesWhat the dashboards show
steadyNormal traffic — and the rollbackflat, healthy, ~40% fill rate
error_burst3 in 10 requests are malformederror ratio climbs; 422s attributable to the caller
traffic_surgeTen times the request ratethroughput and latency climb together — load, not fault
bad_config_deployRetargets every campaign to a dead countryfill rate falls off a cliff
budget_runawayInflates one campaign's daily budgetpacing stops throttling; one campaign takes everything
ADR-0005 — the most important decision so far

Real failures, never simulated ones

No service contains a branch that knows it is being tested. There is no CHAOS=true flag, no debug endpoint, no fake data written into the metrics.

A bad config deploy genuinely sends PATCH requests that rewrite every campaign's targeting. The decision service then reads that configuration and correctly declines to serve. Nothing is faked — the code behaves perfectly against configuration that is wrong.

Which is exactly what a real outage looks like, and exactly what Level 3's agents must be able to diagnose.

flowchart LR
    OP["POST /scenario<br/>bad_config_deploy"] --> SIM["traffic-simulator"]
    SIM -->|"PATCH every campaign<br/>targeting → Antarctica"| CS["campaign-service"]
    SIM -->|"keeps requesting ads"| ADS["ad-decision-service"]
    ADS -->|"reads the changed config"| CS
    ADS --> NF["targeting_mismatch<br/>fill rate 43% → 2%"]
    NF --> PROM["Prometheus"]
    PROM --> GRAF["Grafana<br/>the incident, visible"]
    GRAF -.->|"Level 3"| RCA["ops agent finds<br/>the real cause"]

Because the damage is real, recovery has to be real too. The steady scenario doesn't set a flag back — it restores the original configuration. Which is precisely the remediation a Level 3 agent should recommend.

The first published number

The Level 0 quality gate: eleven automated checks against the running system. Five health endpoints, one metrics-collection check, and one per failure mode. It exits successfully only if all eleven pass, and it leaves the platform healthy.

Its decision logic is pure and separately tested; the live run only supplies numbers. That's what makes it trustworthy in both directions — it cannot pass with the system switched off, and it cannot fail because a test fixture drifted.

Result: 11/11 — 100%. The first real number on the scoreboard.

What reality found — three defects on day one

A $2,147 ceiling nobody knew existed

Budgets were stored in a 32-bit integer column, which caps at 2,147,483,647 micros — about $2,147 of daily budget. The runaway scenario tried to exceed it and Postgres refused.

And it surfaced as a bare 500, violating the project's own Day 2 standard. Now a typed 422, with a handler ensuring no column's range error can ever return untyped again.

The seeding guard was all-or-nothing. One leftover test campaign suppressed the entire seed set, and the simulator drove all its traffic at a single narrowly-targeted campaign — a 2% fill rate with 828 rejections. Seeding is now idempotent per campaign.

And the gate failed itself on its first run. The traffic-surge check read its baseline from the response of the call that caused the surge, so it compared 20 requests/second against 20 requests/second. Caught, fixed, re-run clean — which is what a gate is for.

Level 0 complete: 141 tests, five services, real traffic, real incidents.

Day 06 · Document ingestion

The substrate becomes something the AI knows

Level 1 begins. Everything built so far — the source code, the five decision records, the daily logs, the runbook, the API contracts, the configuration — stops being the thing under construction and becomes the thing the AI knows about.

This is the context layer: give an AI your team's actual accumulated knowledge so it answers from your decisions rather than from generic internet training.

flowchart LR
    RULES["14 ordered rules<br/>what belongs in the corpus"] --> DISC["discover<br/>63 files claimed"]
    APPS["4 FastAPI app objects<br/>app.openapi()"] --> SPECS["4 API contracts<br/>generated, not scraped"]
    DISC --> LOAD["load<br/>title · component · attributes"]
    LOAD --> HASH["normalise line endings<br/>sha256 content hash"]
    SPECS --> HASH
    HASH --> UP["store<br/>added / updated / unchanged"]
    UP --> PRUNE["prune<br/>orphans removed"]
    PRUNE --> CORPUS["corpus.jsonl<br/>67 documents · 550 KiB"]
ADR-0006

A reviewable list, not a crawl

The lazy approach is to walk the folder and take everything not excluded. It works on day one and rots quietly: add a directory of scratch notes and the AI starts grounding its answers in them, with nobody reviewing a change that was never made.

Instead, what belongs is fourteen ordered rules in code. Adding a directory to the corpus is a code change with a test. "What do the agents know" becomes reviewable in a pull request.

The blank decision-record template is deliberately excluded. It's a form, not a decision — grounding an agent in it teaches it to answer with blanks.

Contracts read from the code, not the network

Each service publishes an API contract. The obvious way to collect them is to ask the running services. That would mean the corpus depends on which containers happen to be up, can't be rebuilt automatically, and quietly differs between machines.

Instead the contracts are generated by loading the code and asking it directly. Same bytes on every machine, including one with Docker switched off.

Fingerprints, so nothing is done twice

Every document gets a content hash — a fingerprint that changes if a single character changes. Re-run the pipeline on an unchanged repository and the answer is "67 unchanged, nothing written." That's what makes Day 7's embedding step affordable to repeat.

Every document also carries provenance: where it came from, what kind of thing it is, which part of the system it describes. The ID is the file path itself, so a citation is something a human can paste into an editor.

What reality found

Line endings would have broken everything, silently

Windows ends lines with two invisible characters; Linux uses one. Git warned about the conversion on the very first commit.

Without normalising before fingerprinting, a file hashed on Windows and re-hashed on a Linux CI server would produce different fingerprints for identical content. Every run would report all 67 documents as changed, every embedding would be recomputed, and "unchanged" would mean nothing at all.

Day 6 ended at 201 tests, 67 documents in the corpus.

Day 07 · Chunking, embeddings, vectors

Teaching it to find things by meaning

A whole document is the wrong unit. One API contract is 851 lines — handing an agent all of it to answer one question is useless. Two problems to solve: cut documents into useful pieces, and make those pieces findable by meaning rather than keyword.

Cutting where the content already has seams

The naive approach cuts every 2,000 characters, which slices functions in half. Instead each kind of document is cut where it already has natural boundaries:

flowchart TB
    DOC["A document"] --> R{"What kind?"}
    R -->|"Python"| PY["Split at functions and classes<br/>anchor: the symbol name"]
    R -->|"Markdown"| MD["Split at headings<br/>anchor: the heading path"]
    R -->|"API contract"| OA["Split per endpoint<br/>anchor: POST /campaigns"]
    R -->|"anything else"| WIN["Fixed window<br/>anchor: none"]
    PY & MD & OA & WIN --> CAP["Cap oversized pieces<br/>keeping the anchor"]

Every piece keeps an anchor — the name of the thing it is. ADR-0005 › Alternatives considered. POST /ad-request. A piece that cannot name itself can be returned but not defended.

What an embedding actually is

An embedding turns text into a long list of numbers positioned so that similar meanings land near each other. "Frequency capping" and "how often a viewer sees an advert" share no words — but a good embedding puts them close together. That's how you search by meaning.

Different models do this with different quality. So rather than picking one, BELLWETHER runs four behind a single interchangeable interface.

ADR-0007 — the original plan was backwards

Hosted models by default, local for free iteration

The Day 0 plan said run a model locally to save money. Measuring the corpus inverted that entirely.

The whole corpus is ~149,000 tokens — about two cents to embed with the best model available. Meanwhile running locally means installing PyTorch: 2.5 GB on disk and a resident process competing for memory with Docker on a 16 GB laptop with no graphics card.

The best option turned out to be the lightest one locally, because the computation happens on someone else's hardware. The premise flipped the moment anyone measured.

Trigger to revisit: a corpus past a few million tokens, where per-run cost stops being a rounding error.

ADR-0008

One point per piece, one vector per engine

Comparing engines is only meaningful if each one sees identical input. The vector database stores each text piece as a single record carrying all four engines' vectors at once, alongside one shared set of provenance.

Switching engines becomes a query parameter. Fairness stops being a discipline someone has to remember and becomes a property of the storage layout.

What two cents buys

Asked: "how often can the same viewer be shown one advert?" — a question containing none of the words frequency, cap, member, or impression:

EngineScoreBest match
gemini0.765Substrate gaps › SG-04 › Multiple frequency-cap windows
potion0.428ad_decision_service.main.decide
hashing0.328an unrelated plan section — noise, as designed

Gemini understood the question. The local model found the right neighbourhood. The keyword-only baseline found nothing, because no word in the question appears in the answer. That gap is the entire argument for paying for embeddings — measured here, not quoted from a vendor.

What reality found — the most dangerous bug yet

The database silently destroyed data, and every test passed

Writing one engine's vector to a record replaced the whole record. So storing Voyage's vector silently deleted Gemini's.

Every unit test passed — because the test double faithfully reproduced the wrong mental model. The comparison would have rendered a beautiful chart built on destroyed data.

It was caught by pointing the code at a real running database and asking one question: is the first vector still there? It was not.

Two more: a text file written by PowerShell carried an invisible marker that turned VOYAGE_API_KEY into an unrecognisable name — presenting as a missing key while the file plainly contained one. And a verification command was piped through another program, which swallowed its failure signal, so nine type errors were committed while appearing green.

Day 7 ended at 289 tests, 585 pieces, three engines measured.

The pattern worth noticing

Read the eight days back to back and the same shape repeats. A decision gets made and written down with the condition that would reverse it. Then reality tests it — and in every single case, running the real thing found something the plan had not.

  • A metric collision that only appeared when two services loaded together
  • A dashboard connection ID that changed on restart
  • A $2,147 budget ceiling nobody knew existed
  • A quality gate that measured itself and passed
  • Line endings that would have made every fingerprint meaningless
  • A database that quietly deleted data while every test agreed it was fine

None of these were found by careful thinking. All of them were found by running the real thing against real conditions and looking at what happened.

Which is, in the end, the entire argument of the project. An AI that writes code fast is only useful if something can prove the code is right. That proof cannot come from a fixture, a mock, or a demo that worked once. It has to come from the real system, misbehaving in real ways, measured by numbers published whether they flatter or not.
