# ADR-0004: Idempotent on the primary key, aggregated on read

**Date:** 2026-07-21
**Status:** Accepted

## Context
event-service turns the serving path into a durable record: every impression and click the platform delivered. Two questions had to be answered before writing a line of it.

**How does a retry not become a second impression?** Delivery reports are fire-and-forget over a network. A client that times out waiting for the ack has no way to know whether the event landed, so it retries — and on Day 5 the traffic-simulator will do exactly that, deliberately. An impression counted twice is a campaign billed twice.

**Where do the delivery numbers come from?** Impressions, clicks, CTR, and spend per campaign are read constantly — by the Grafana dashboards, and three levels from now by an RCA agent asking what a campaign actually delivered during an incident window.

## Decision
**Ingestion is idempotent on a caller-supplied `event_id`, enforced by the primary key.** The client chooses the id; `ad_events.event_id` is the primary key; the insert either wins or raises `IntegrityError`. The second outcome is a `200` ack with `status: "duplicate"` — not an error, because from the caller's point of view the event *is* recorded.

**Delivery is aggregated on read** — one `GROUP BY campaign_id` over `ad_events`, computed per request. There is no rollup table and no materialized view.

## Alternatives considered
- **Check-then-insert (`SELECT` by id, insert if absent):** rejected. It races. Two concurrent reports of the same impression both read "absent" and both insert. The primary key does not race — that is its entire job.
- **A dedupe window in Redis:** rejected. It adds a second store to the write path to protect a constraint Postgres already enforces for free, and a Redis flush would silently re-open the double-count.
- **Server-generated event ids:** rejected outright. It makes idempotency impossible — every retry is a new id and a new impression by construction.
- **A rollup table written on ingest:** rejected for now. It is a second copy of the truth, and the two drift the first time a backfill or a delete happens. At Level 0 volumes the `GROUP BY` is trivial.
- **Deleting duplicates in a nightly job:** rejected. It means the numbers are wrong all day, every day, until the job runs.

## Consequences
- A caller that does *not* send an `event_id` gets one generated by the schema default and forfeits idempotency. That is the caller's choice to make, and the API documents it.
- The `ad_events` table only grows. Nothing prunes it yet; at Level 0 volumes that is fine, and Day 5 is what will tell us when it stops being fine.
- `/delivery` scans the events table. It is fast because the table is small, not because the query is clever — an honest constraint, not an optimization.
- Spend recorded here is still an estimate at a flat per-impression price (ADR-0003), not billing. What changed is that it is now *durable and auditable*: there is a row per impression behind every micro.
- event-service creates and reads **its own table** in the shared Postgres database and touches nobody else's. ADR-0002's Alembic trigger — "the first time a second service reads these tables" — remains unfired.

**The trigger that flips this decision:** Day 5's traffic-simulator under sustained load. When `/delivery` p95 crosses ~200 ms, or the events table outgrows a single day of retention, a rollup table (or a scheduled materialized view) becomes worth the copy-of-the-truth it costs.
