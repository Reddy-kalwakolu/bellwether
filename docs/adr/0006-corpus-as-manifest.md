# ADR-0006: The corpus is a manifest, and service contracts are generated, not scraped

**Date:** 2026-07-21
**Status:** Accepted

## Context
The context layer grounds every agent from Level 2 onward. Whatever is in the corpus is what an agent will believe, quote, and cite; whatever is missing is a question it will answer from the model's priors instead of from this team's decisions. So two questions have to be answered deliberately rather than by convenience: **what gets in**, and **where the API contracts come from.**

Both have an obvious lazy answer. Walk the repo and take everything that is not excluded. Fetch `http://localhost:8001/openapi.json` from the running service. Both work on the first day and quietly rot afterwards.

## Decision
**The corpus is an ordered list of rules in code** — `CORPUS_RULES` in `bellwether/context/discovery.py` — each pairing a glob with the source type it admits. Exclusions exist, but as a safety net under the rules, not as the mechanism. First rule to claim a path owns it.

**The four OpenAPI contracts are generated by importing the FastAPI app objects and calling `app.openapi()`.** Ingestion opens no socket and needs nothing running. `json.dumps(..., sort_keys=True, indent=2)` makes the text stable, which makes the content hash stable, which is what lets "nothing changed" mean something.

Two supporting rules fall out of the same reasoning. **Content is normalised to `\n` before hashing**, so a Windows checkout and a Linux CI runner agree on what a document is. And **the pipeline prunes**: a document whose source has left the repo leaves the corpus, because an orphaned document is worse than a missing one — an agent will happily cite it.

## Alternatives considered
- **Crawl everything, minus an exclusion list:** rejected. It grows silently. Add a directory of scratch notes, a vendored dependency, or a generated report, and the context layer starts grounding agents in it with nobody reviewing a diff. A rule list makes "what do the agents know" a reviewable code change with a test attached.
- **`curl` the running services for their OpenAPI specs:** rejected. The corpus would depend on which containers happened to be up, could not be built in CI, could not be built on a laptop with Docker stopped, and would differ between machines. A context layer that is not reproducible is not a foundation.
- **Commit the generated specs as files in the repo:** rejected. A second copy of the contract that goes stale the moment a route changes, and a diff on every PR that touches an endpoint. The app object already *is* the contract.
- **Content-addressed document ids (a UUID or a hash):** rejected. `doc_id` is `docs/adr/0005-real-failures-not-mocked.md` because a citation has to be something a human can paste into an editor. Level 1's promise is that every agent answer names its source; an opaque id would break that at the last inch.
- **Ingesting `bellwether/` itself:** deferred, not rejected. The corpus is Level 0's output. Whether the AI layer should read its own source is a real question — it belongs to Day 9's knowledge graph, taken as a decision, rather than arriving as the side effect of a glob.

## Consequences
- Adding a directory to the corpus is a code change with a test, which is the accountability property this whole level is for.
- Ingestion imports all four service apps, so an import-time side effect in any service becomes an ingestion-time side effect. Today there are none — the services build their clients lazily and open no connections at import — and that is now a property the pipeline depends on.
- The store sits behind a `DocumentStore` protocol with a JSONL implementation. Day 7's vector store arrives behind `upsert`, and `pipeline.py` does not change.
- Content hashing makes re-ingestion nearly free: the second run of an unchanged repo reports 65 unchanged and writes no new versions. That is what makes re-running embeddings affordable once Day 7 attaches a cost to every changed document.

**The trigger that flips this:** when the corpus outgrows what a rule list can describe — external documentation, ticket systems, Slack threads, anything not in this repo — the in-code tuple becomes a manifest file with the same semantics. The manifest stays; only its location moves.
